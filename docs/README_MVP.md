# MVP: менеджер LaTeX-проектов

Цель MVP - быстро получить рабочую оболочку вокруг текущего `vkr.cls` и шаблонов преподавателя.
Оформление считаем уже решенным: оно задано классом, `setup.tex` и переменными в `main.tex`.

## Что есть сейчас

- `vkr.cls` - основное оформление: поля, шрифты, интервалы, титульники, содержание, рисунки, таблицы, список источников.
- `setup.tex` - подключение пакетов и настройки подписей, таблиц, листингов.
- `main.tex` - точка входа документа, где выбирается тип отчета флагами `\ВКРtrue`, `\Практикаtrue`, `\Курсоваяtrue`.
- Набор `.tex`-разделов для ВКР/курсовой/практики.
- `makefile` с локальной сборкой через `xelatex`.

Лабораторной работы как отдельного режима пока нет.

## Что должен дать MVP

1. Проекты на диске.
   Пользователь создает проект, а приложение создает обычную папку с файлами. Эту папку может читать и редактировать ИИ-агент.

2. Несколько документов внутри одного проекта.
   Например, в одном проекте могут лежать `lab-01`, `lab-02`, `practice-report`, `course-paper`.

3. Быстрое создание документов из шаблонов.
   При создании выбирается тип: `course`, `practice`, `lab`. Приложение копирует нужный шаблон и заполняет стартовые переменные.

4. Build API.
   ИИ-агент или UI дергает HTTP endpoint, backend запускает `xelatex` в Docker/TeX Live окружении и сохраняет PDF в папку документа.

5. Просмотр в браузере.
   После сборки UI показывает PDF в iframe/встроенном просмотрщике и рядом выводит лог сборки.

## Главный принцип

Файлы важнее базы данных. База, если появится, должна быть индексом и кешем, а не единственным источником правды.
Источник правды для ИИ и студента - физическая структура проекта на диске.
